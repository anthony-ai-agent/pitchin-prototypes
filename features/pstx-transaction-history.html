Header filters, not a segmented control. Event and Status are filtered the same way the rest of the table already works — a "▽" trigger embedded in each column header opens a checkbox menu (All / Clear / Apply). ECF investment and Refund just need adding to the existing Event-filter option list; Status reuses the existing Pending / Approved / Completed / Failed set untouched.
Counter → "Counter / Campaign". Same column, same link styling as a PSTX ticker — for these two events it renders the campaign name instead of a ticker and routes to the campaign page (/equity/{{slug}}, same pattern as My Investments) instead of /pstx/{{ticker}}.
Balance on Hold is unchanged from Phase 2.1 — same three-bucket drillable card (PSTX Open Orders / Withdrawal Requests / ECF investment). It isn't re-specified here; this screen shows it simply reflecting fewer/no held ECF pledges once they've allotted or been refunded.
Money-sensitive (backend, one transaction): allotment debits the wallet and releases the ECF hold; a refund credits the wallet and releases the ECF hold — same invariant, opposite sign. Front-end re-reads wallet-details; no client-side maths. Verify on real Postgres (ADR-0005).
The 2.2 / 2.3 tags & row tint above are prototype annotations only — they do not ship. The campaign-link click in this prototype just shows where it would navigate (real build uses routerLink).
